# 데이터 손실 방지(DLP) 운영 전략 (Data Loss Prevention)

본 문서는 TogetherChise 플랫폼의 데이터 손실을 원천적으로 차단하기 위한 승인된 운영 전략을 기술합니다.

---

## 🛡️ 1. 핵심 보존 방안

### 💎 사용자 및 비즈니스 데이터 (Firestore)
- **PITR (Point-in-Time Recovery) 활성화:**
  - **설명:** 데이터베이스 상태를 최근 7일 이내의 특정 시점(분 단위)으로 복구할 수 있는 기능입니다.
  - **용도:** 운영자 실수로 인한 데이터 대량 삭제 또는 로직 오류 시 즉각 복구.
- **데일리 스케줄 백업 (Scheduled Export):**
  - **설명:** 매일 1회 Firestore 전체 데이터를 Google Cloud Storage(GCS)로 내보냅니다.
  - **보관:** 다중 리전(Multi-region)에 저장하여 장기 보존 및 재해 피해 최소화.

### 💻 소스 코드 및 설정 (System Data)
- **GitHub 원격 저장소 이중화:**
  - 모든 코드는 `main` 브랜치에 안전하게 push되어 있으며, local 환경 손실 시 즉시 재구축 가능합니다.
- **환경 변수(.env) 보안 관리:**
  - 로컬 `.env.local` 외에도 승인된 보안 관리 도구에 설정값을 백업하여 유실에 대비합니다.

### 🖼️ 미디어 및 증빙 자산 (Cloud Storage)
- **객체 버전 관리 (Object Versioning) 활성화:**
  - 동일한 이름으로 파일을 업로드하거나 삭제하더라도 과거 버전을 유지하여 실수로부터 보호합니다.

---

## 🔄 2. 복구 및 검증 프로세스

- **분기별 복구 드릴:** 3개월마다 1회 백업 데이터를 테스트 환경에 복구하여 실제 가용성을 확인합니다.
- **실시간 모니터링:** 백업 실패나 비정상적인 데이터 삭제 감지 시 관리자에게 즉각 알림을 발송합니다.
- **최소 권한 원칙 (PoLP):** 데이터 삭제 권한은 최소한의 인원에게만 부여하고, 모든 삭제 작업은 로그에 기록합니다.
- **네트워크 레이턴시 보호:** 클라이언트 측 **타임아웃(Timeout)** 로직을 통해 불완전한 데이터 쓰기 시도 시 사용자에게 즉각 알림을 주어 중복/손상 데이터 생성을 방지합니다.
